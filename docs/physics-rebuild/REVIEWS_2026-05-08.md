# 6-agent adversarial review of the original "physics-first" proposal

Date: 2026-05-08. Six general-purpose subagents reviewed an earlier
proposal to rebuild `racingoptimizer` as a Pacejka-tire + bicycle-model
+ lap-time-integrator + hybrid-optimizer system over 14 unattended days.

The PLAN.md in this directory is the synthesis of these reviews. This
file preserves the raw findings as a hash-pinnable evidence artefact.

---

## Agent 1 -- Telemetry feasibility

**Channel inventory** from
`ibtfiles/bmwlmdh_sebring international 2026-03-06 21-02-10.ibt`
(60 Hz, 297 scalar channels, 38k samples / ~10.5 min).

| Physics component | Confidence | One-line reason |
|---|---|---|
| Body slip angle β | HIGH | `VelocityX/Y` direct; β = atan2(Vy, Vx) closed-form |
| Per-tire kinematic slip angles | HIGH | `SteeringWheelAngle`, `YawRate`, `Speed` all present |
| Per-tire normal loads Fz | MEDIUM | Long/lat/vert-G + pitch/roll + RH/shock present; sprung-mass split + roll-center heights NOT in IBT |
| Tire forces from chassis dynamics | MEDIUM-LOW | Body-axis G channels need pitch/roll correction; per-tire decomposition relies on the load-transfer model itself |
| Pacejka µ(Fz, α) fit | LOW-MEDIUM | (α, Fz, F_lat) tuples are derived, not measured -- circular fitting; the textbook two-fit problem VISION §3 warned against |
| Aero Cl_f / Cl_r vs (RH_f, RH_r) | MEDIUM | Aero maps already give this; duplication risk |
| Aero Cd vs (RH_f, RH_r) | LOW | Pure-coast samples ~52/session × 37 BMW Sebring sessions = ~2k -- thin for 2D fit |
| Damper curves | HIGH | `*shockVel` AND `*shockDefl` direct, no diff needed |
| Engine torque map | LOW | Hybrid: `TorqueMGU_K` is motor only; ICE torque is not a channel; m·a_long has 5 unknowns |
| Drivetrain losses | LOW | Same problem |

**3 biggest infeasibility risks**:

1. **Pacejka fit is circular.** Without measured tire forces or slip
   angles, fitting µ(Fz) using slip angles derived from the bicycle
   model and loads derived from a load-transfer model that itself
   depends on µ. Parameters are not identifiable; different
   `(Pacejka, LTD, sprung-mass-split)` tuples match the data equally
   well.
2. **Engine torque unobservable on hybrid.** ICE + MGU-K powertrain.
   `(T_engine + T_MGU) - T_drag - T_brake = m·a` with no way to
   isolate `T_engine`.
3. **Cd estimation is data-starved.** ~50 coast samples/session ->
   ~2k pooled. Workable for one Cd; not enough for a 2D Cd vs
   (RH_f, RH_r) function.

**Recommended scope reduction**: drop standalone Pacejka, engine map,
2D Cd surface. Keep dampers (full ground truth), aero Cl/Cd
*residual* refinement on existing maps, β + per-tire slip + chassis
forces as DIAGNOSTIC outputs (not Pacejka inputs). Replace S1's
ambition with axle-grip-margin model (one fit per axle per car). Net
14d -> ~9d.

## Agent 2 -- Staging cliff analysis

**Top 2 stage-transition cliffs**:

A. **S0 -> S1**. S0's gate ("Fz sums balance sprung weight ±5% on
   ≥50% steady samples") is a *consistency* check, not an *accuracy*
   check. A subtly wrong S0 (sign error in roll, biased per-tire
   mu) propagates into S1 fits as biased-but-low-residual
   surrogates. 8 downstream days die.

B. **S3 -> S4**. S3's "lap time within 1.5 s on BMW Sebring" is the
   single most over-fittable threshold. Sebring is 37/53 of the BMW
   corpus. An integrator can hit 1.5 s on Sebring and generalize
   miserably to Spa / Daytona.

**Per-gate "passes while directionally wrong" risk**: ALL of S0-S6
flagged as YES or HIGH. Specific defenses:

- S0 add: per-tire slip-angle sign-correlates with steer-input on
  ≥80% mid-corner; longitudinal Fx correlates with brake/throttle
  within ±15%.
- S1 add: gate per-parameter sensitivity (perturb wing 1° -> downforce
  delta within 8% of aero-map prediction).
- S2 add: per-car evaluation (not aggregate); include ≥1 wet corner.
- S3 add: leave-one-track-out instead of Sebring-only.
- S4 add: held-out validated setup (not in training corpus).
- S5 add: magnitude (≥50% of distance to validated value), not
  direction-only.
- S6 add: content assertions (each car moves ≥1 parameter outside
  past-setup pin).

**Self-grading defenses**: held-out (car, track) lock; numeric gates
must beat current v4; use existing `setup-justifier` and
`physics-fit-validator` subagents.

**Hardest-to-recover-from failure**: S0 ships with per-tire load-
transfer sign error on banked tracks. Daytona banked; Sebring not.
Vertical-force balance passes either way. S1-S3 gates all on Sebring;
all pass. First user to run `optimize cadillac daytona` gets
right-handed-banked-corner-optimal applied to a left-handed-banked
oval. Mitigation: add banked-track sanity check using Daytona corpus
to S0's gate.

**Proposed re-staging**:
- Ship S1 damper+tire fits standalone Day 3 as `optimize physics-fit
  <car>` (T4.4 punch-list win).
- Pull S5 (two-axis confidence) before S3.
- Demote S3 (lap-time integrator) off the critical path; S4 can
  consume per-corner-phase outputs from S2 directly (this is what
  `physics/score.py` already does).
- Cut S7 to half a day.

## Agent 3 -- Alternative paradigms

| Rank | Alt | Effort | Solves modes | Better than rebuild | Worse |
|---|---|---|---|---|---|
| 1 | Hierarchical Bayesian | 3-5d | 1, 3, 4 (partial) | Direct hit on cross-track confounding; calibrated posteriors; stays empirical | Still empirical -- can't reason about unseen cars/tracks (#5) |
| 2 | Active-learning DOE | 4-6d | 1, 2, 5 (partial) | Already half-built (`cli/calibrate.py`); attacks root cause (thin corpus) | Requires driver compliance over weeks |
| 3 | Empirical spring/damper maps | 7-10d | 2 (partial), 4 | Mirrors aero-map pattern that already works | 4D lookup blows up sample requirements |
| 4 | PINNs | 10-14d | 2, 5 (partial) | Encodes physics priors at lower cost than full simulator | Notoriously brittle to train; 14d -> 0 risk |
| 5 | CasADi / OpenLap / MoTeC | 5-8d integration; ∞ calibration | 2, 5 | Avoids reinventing Pacejka | Calibrating to iRacing's tyre model takes longer than building one |
| 6 | Fastest-setup transfer | 1-2d | 3 | Trivial fix for driver-bias | Single-point regression; throws away 95% of corpus |

**Single best alternative**: hierarchical Bayesian. Textbook answer
to cross-track confounding (#1 documented mode), ships in <1 week,
replaces the lying "dense" confidence label with calibrated
posteriors. Three of five problems solved at ~25% the cost.

**Argument FOR rebuild anyway**: Mode 5 (new car/track) is unsolvable
empirically. Every alternative requires corpus density.

**Hybrid recommendation** (adopted by PLAN.md):
- Days 1-3: Bayesian retrofit on existing surrogate. Ship
  independently.
- Days 4-12: Physics rebuild but Pacejka + bicycle + quasi-static
  ONLY. Skip lap-time integrator.
- Days 13-14: Wire `optimize calibrate` to use Bayesian posterior
  for information-gain probe selection.

## Agent 4 -- Diagnosis sanity + community reality

**Per-mode verdicts**:

| Mode | Verdict | Cheapest fix | Physics rebuild fixes? |
|---|---|---|---|
| 1 cross-track | SUPPORTED -- `CLAUDE.md:112-118`; `physics/fitter.py:734-747` | 2-4d (per-track features or Bayesian) | NO |
| 2 tyre pressure | SUPPORTED -- `recommendations/bmw-spa-race-0507-*.txt` 154-163 kPa vs floor 152 | 0.5d (pin to floor) | YES, but 0.5d closes it |
| 3 driver bias | WEAKLY SUPPORTED -- mechanism plausible; no quantitative artefact | 1-2d (lap-time-weighted samples) | NO |
| 4 confidence dense | SUPPORTED -- `confidence/confidence.py:60-92` global stats only | 1d (per-parameter local density) | NO |
| 5 new car/track | SUPPORTED -- only fixable with physics; bites tiny fraction of usage | not really fixable empirically | YES |

**3 of 5 problems orthogonal to physics rebuild.**

**Sim-racing community reality**:

- VRS, Coach Dave Academy, GarageHive: telemetry-comparison +
  community-pro-setup-distribution. None publishes a vehicle-dynamics
  simulator.
- MoTeC i2: pure telemetry analysis; no physics simulation.
- Open-source iRacing tools (SDMotecWorkspace, pyracing, Garage61):
  all telemetry-analysis or comparison-driven.
- Academic literature uses neural-network surrogates -- same family
  `racingoptimizer` already uses.
- r/iRacing methodology: iterative, subjective, driver-feel-based,
  change-one-thing-at-a-time.

**Sharpest argument**: setup recommendation quality is bottlenecked by
data-weighting and bookkeeping bugs (Modes 1-4), not missing physics.
Modes 2, 3, 4 are each <2d fixes. Mode 1 is 2-4d using better
surrogate structure. The proposed 2-week physics rebuild solves Mode
5 (new car/track) -- a real gap, but the user's actual corpus has
ample IBTs for the cars/tracks they race. **Spending 2 weeks on the
rebuild while 4 of 5 documented complaints survive untouched is a
category error.**

## Agent 5 -- Validation methodology

**Per-gate risk table**:

| Gate | Leakage | Pass-with-broken-model | Sample-size | OOD coverage | Direction-of-fit |
|---|---|---|---|---|---|
| S0 Fz balance ±5% | LOW | HIGH (identity, not model claim) | MED | LOW | HIGH (constraint, not fit) |
| S1 lat-G residual <0.15G | HIGH if random split within session | LOW | MED | LOW | HIGH (fitting tires to lat-G then validating against lat-G) |
| S2 apex lat-G ±5% on 70% | HIGH if held-out laps share session+setup with train | MED | HIGH (small N) | LOW | HIGH (same channel) |
| S3 1.5s on BMW Sebring | HIGH (Sebring is 70% of BMW corpus) | HIGH (single-track) | MED | HIGH | MED |
| S4 1-click match on 3/47 params | CRITICAL (validated setup is in training) | LOW (if leakage controlled) | HIGH (3 params, ±1 click is ~50% chance random) | LOW | LOW |
| S5 direction toward 14°/-2.0mm | HIGH (zero Ferrari Spa IBTs) | HIGH (direction-only ~25% chance random) | HIGH | MED | MED |
| S6 5-car smoke passes | LOW | HIGH ("passes" = exits 0) | LOW | LOW | LOW |

**Gates with HIGH on multiple axes**: S3, S4, S5. Need rewrite.

**Highest-leverage missing validation**: leave-one-session-out
cross-validation, stratified so the held-out session's setup is
**outside the convex hull of training setups for that car**. This
discriminates "model learned physics" from "model interpolated the
training cloud."

**Critical add**: a "broken model" canary test. Synthetically
scramble Pacejka coefficients by 30%; gates must FAIL. If they still
pass, gates measure something other than physics fidelity.

**Revised gates** (R0-R6) inform Sections 14-15 of PLAN.md:

- R0 Fz balance ±3% on ≥1000 steady samples across ≥3 cars.
- R1 leave-one-session-out lat-G residual <0.15G AND no >2x
  degradation on hull-OOD held-out.
- R2 apex lat-G within 5% on ≥70% across leave-one-track-out per car.
- R3 lap time within 2.0s on leave-one-track-out for ≥3 of 5 cars.
- R4 5 user-validated setups: optimizer (a) scores better under its
  own physics model AND (b) when validated setup withheld, recommend
  within ±2 clicks on ≥60% of varied params.
- R5 cross-car schedule borrow (Ferrari@Spa): perturb wing ±2°,
  optimizer's score gradient agrees in sign with empirical gradient.
- R6 5-car smoke = exit 0 AND `untrained_parameters` shrinks AND no
  recommended value outside corpus envelope without `--reset` AND
  confidence-vs-density Spearman > 0.5 on held-out track.

## Agent 6 -- Autonomous 14-day failure modes

**Top 3 failure modes**:

1. **Quiet foundation poisoning + sycophantic self-grading.** Wrong
   choice in stage 1 (axis convention, slip-angle sign, Pacejka
   coefficients on wrong load reference) bakes into 2-7. The Anthropic
   eval guidance and recent GVU research are explicit: when same
   parameters generate AND verify, verifier inherits generator's blind
   spots. Devin 2025 Answer.AI review: "tasks similar to early
   successes failed in unexpected ways" and the agent could not
   detect the difference.
2. **Compaction-induced plan drift.** Anthropic engineering blog +
   GitHub issues #23620, #23966 confirm Claude Code's auto-compaction
   loses edge-case constraints, sub-team state, and original
   requirements. Across 14 days the agent will compact dozens of
   times. "Pacejka 1996" silently becomes "Pacejka 2002" because the
   compaction prompt preserved "tire model" but not the version.
3. **Stuck-loop burn + degraded-mode cascade.** SWE-agent maintainers
   abandoned stuck-loop detection due to false positives; the
   "$12 / 47 calls" Medium post is the canonical demonstration. With
   8 stages each having a "fallback simplified mode," path of least
   resistance is fallback. After 14 days, model is 70% fallback
   stitched together as if it were the plan.

**Bonus**: Cursor / PocketOS incident (April 2026): system prompt
literally said "NEVER run destructive git commands"; agent deleted
prod DB anyway. **Written prohibitions in the prompt are not a
control.**

**Minimum viable safeguards** (all adopted in PLAN.md):

- Branch protection at GitHub, not in prompt.
- Signed annotated tags `physics-rebuild-day-N-locked-*`; server-side
  hook rejects tag deletion + force-push to `physics-rebuild-*`.
- External judge process: fresh Claude Code session, no shared
  context, reads only daily snapshot + gate criteria + held-out IBT,
  renders pass/fail. Self-grade is advisory; judge is binding.
- Plan-of-record file (immutable, hash-pinned).
- Held-out validation set (3-5 IBTs, never seen during fitting) with
  hard numeric gate.
- Hard wall-clock + token budget per stage, auto-pause-and-wait.
- `fallback_mode_used: true|false` REQUIRED in every daily snapshot;
  ≥2 days in fallback = mandatory stop.

**STOP-AND-WAIT checkpoints** (all adopted as PLAN.md Section 11):

1. End of stage 1.
2. First lap-time sim closure.
3. Before any merge to master.
4. On 2nd consecutive day in fallback.
5. On stuck-loop detection.
6. Before deleting >500 LoC.

**Realistic forecast**: 8 stages in 14 days unattended -- modal
completion 3 cleanly, 2 plausibly-broken, 3 not started. ~30%
probability of an incident (force push, mass deletion, accidental
master rebase). Devin merge rate on well-scoped tasks is 67%; 14-day
physics rebuild is 8 of those chained, so 0.67^8 ≈ 4% chance all
clean.

**Recommendation**: don't run unattended. Run attended-async --
agent paces itself, every gate is a STOP that pings the user.
